## WHY the roster was respawned FRESH (not resumed)

`claude-chat -r <name>` is just `claude -r <name>` (`.bashrc:74`). The friendly
name → Claude-session-id resolution **drifts** across compaction/restart, so a
window keeps its correct title but binds to a *different* underlying session,
which the daemon labels under *yet another* name. Today's proof: the `observer-1`
window was running session `ebf8b668` — which the daemon called **agent-4** —
already in the old chat under that wrong identity, while the id `session_list`
labeled "observer-1" (`43267171`) was a ghost in no window. The current roster's
transcripts were **already cross-linked**; the fixes harden resume *going
forward* but can't un-scramble an already-corrupted roster. A fresh `roster`
(`-n`) creates clean sessions with correct title↔id↔name from launch. **Use
`roster` (START), NOT `roster resume`, until the resume-index resolution is
fixed** (see `.bashrc:83-86` note).

NOTE: roles bind by NAME, not window/layout order — `do_launch` sets
`--title="$name"` + `KHIMAIRA_ROSTER_SLOT=$INSTANCE:$name` + a TRAP-2 token file
(window-id → slot, read by path). The layout (grid/hub/splits) is cosmetic. The
scramble is purely the resume name→session-id drift, not positional.

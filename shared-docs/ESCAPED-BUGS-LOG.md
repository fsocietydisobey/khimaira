## 1. Meta-class

**Green unit suite (or static config), broken live behavior — because the bug lives in a SEAM the unit tests mock past, or a PREMISE the static check never verifies at runtime.** Every escape below passed N/N green (or "rule exists in config") while the live path was broken. The tests aren't wrong; they're blind to the seam *by construction* — they mock the exact boundary where the defect lives (the real event a producer emits, the real payload, the real enforcement hook, the real runtime input).

**The meta-class is RECURSIVE and extends to GOVERNANCE, not just data flow.** A structural rule (Themis hint, lint gate) can itself escape if its *enforcement mechanism* is absent in some environment — "the rule exists in config" (static green) ≠ "the hook actually intercepts the tool call" (live-enforced). Standing invariant the corpus enforces: **every enforcement mechanism (test OR governance hook) must be verified to EXECUTE / INTERCEPT for real — presence-in-config is not enforcement.**

## 2. Escaped bugs

### themis-hook-dormant-standalone [enforcement-premise · environment-config]
- **Symptom (live):** a standalone-spawned agent session ("void", spawned by Joseph outside the roster bootstrap) ran `git stash` and `git checkout` on its own edits — state-changing git that the Themis rule IN-UNIVERSAL-1 ("agents never run state-changing git", shipped 2026-06-17) should STRUCTURALLY BLOCK. No block fired; the commands executed.
- **Root-cause (audit-grade):** IN-UNIVERSAL-1 is enforced by a Themis **PreToolUse hook** wired into the tool-call path only for sessions bootstrapped through the roster launch path. A session spawned standalone has no PreToolUse hook on its Bash tool-calls, so Themis never sees the `git` invocation — the rule is present in Themis config (static green) but **dormant at runtime** (no interceptor on this session's path). [evidence: audit-grade — the git commands were observed executing successfully in the standalone session that the rule names as forbidden.]
- **Why the test missed it:** the rule's guards (`test_role_convention_lint` + Themis rule-condition unit tests) assert the RULE EXISTS and its condition matches a state-changing-git tool-call — they exercise the rule object, assuming the hook is on the path. No test asserts the PreToolUse hook is actually WIRED + INTERCEPTING for a given session *class*. Static rule-existence ≠ live enforcement; the hook-present premise is unverified.
- **Catching-test:** an **assert-it-enforces** gate — a live probe per session class (roster-bootstrapped AND standalone) that issues a benign state-changing-git tool call and asserts Themis BLOCKS it; plus a startup/CI check that the Themis PreToolUse hook is registered on the tool-call path (premise-present), not merely that the rule is in config. **FORWARD.**

### formatter-premise-vs-file-reality [tooling-premise]
- **Symptom (live):** running `black <file>` on `roster_recovery.py` (the repo's "format every file you modify" rule) churned ~313 lines across the whole file and buried the real 98-line diff — even though the edit was 4 localized hunks. Same on `guard5.py` + `auto_dispatch.py`.
- **Root-cause (audit-grade):** these files carry **intentional manual formatting** (hand-split multi-arg calls, hand-wrapped strings) that NO black line-width preserves — `black -l 88 / -l 100 / -l 120 --check` each reports "would reformat" against the committed file. So the file's committed baseline is not any-black-clean; a blind whole-file format reflows the entire file, not just the new lines. The repo has no enforced formatter config (no `[tool.black]`, no `.pre-commit-config.yaml`) for these files. [evidence: audit-grade — verified `black -l {88,100,120} --check` all reformat the committed `git show HEAD:` baseline.]
- **Why the test missed it:** there is no test — "format every file" is a workflow rule, and it ASSUMES a formatter-clean baseline. The premise (the file is black-clean at the project width) is never checked; on a manually-formatted file the rule is actively harmful (churns the diff, can break intentional layout).
- **Catching-test:** **per-file formatter-width probe** before formatting — `black -l W --check <committed-file>`; if it reformats at every standard W, the file is manually formatted → hand-match the edit's style, do NOT run a whole-file formatter. Durable fix: a `.gitattributes` / pyproject exclude (or a `# fmt: off`-style marker) so CI never reformats these files, making the premise explicit. **FORWARD.**

Green against an assumed input whose PREMISE (gate-tasks exist in prod) is dead. Premise-correct: uses the DIRECT-verdict path real rosters use, not the dormant gate-task wrapper. **ADDED 476f2b5** (test_direct_verdict_obligation.py, 7 tests; verified fires on the 5 real owing cases, 0 overlap with the 10 committable).

### owed-verdict-storm-on-accumulated-backlog [test-state-coverage]
- **Symptom (live):** the moment the watchdog deployed, it STORMED muther's week-old roster — it flagged owed-verdict on 5+ HISTORICAL `done`-not-`approved` tasks (research/audit/spike tasks, done 7h–104h ago, that legitimately skip the approve gate). Reviewers were genuinely idle with nothing CURRENT owed; cooldown capped it to ≤1/5min/window (noise, not damage) but it interrupted live reviewers.
- **Root-cause (audit-grade):** `_get_session_obligations`'s direct-verdict obligation scanned ALL `done` partial-verdict tasks **regardless of age** — the ≥1-verdict gate-existence clause proves "under review" but not "currently/recently under review". A long-lived roster accumulates done-not-approved tasks; on deploy they ALL backfill as "owed" at once. A 5-day-old research task with a stray critic-approve is indistinguishable from a fresh mid-review stall. [evidence: audit-grade — chat-f4951a43cbec, 17 real done tasks, the 5 partial-verdict ones 7h–104h old; with the recency filter live owed=0, with the filter off (huge window) live owed=5 — proving recency, not another cause, gates them.]
- **Why the test missed it:** the storm-safety proof was **ZERO-wakes on a QUIET roster, point-in-time** — it tested a single fresh-state snapshot, never a LONG-LIVED roster with accumulated backlog. The unit fixtures built one fresh task at a time; no test exercised the live state *distribution* (many old done-not-approved tasks). The test's STATE coverage didn't match production's state. Same class shape as detector-premise-vs-runtime, but the dead premise is the TEST FIXTURE's state, not a field.
- **Catching-test:** recency dimension in the contract — `done`-ts older than `KHIMAIRA_OWED_VERDICT_WINDOW_S` (default 3600s) → NO obligation; within window → obligation; no parseable done-ts → skip. Pull done-ts from the status-TRANSITION event (the `task_update` ts that set `done`), not create-ts. Validate against a real long-lived roster's chat, not just a constructed quiet one. **ADDED** (test_direct_verdict_obligation.py +3 recency tests; live-validated 5→0 on chat-f4951a43cbec). General lesson: a zero-wakes/zero-fires proof must cover ACCUMULATED state over time, not just a clean point-in-time snapshot.

### wake-targets-window-by-title-not-session-identity [enforcement-premise · detector-premise]
- **Symptom (live):** a chat-member agent (`void`, joined chat-3e677725c16e with role=agent) could NOT be auto-woken by the daemon — the idle-but-owing watchdog never reached it; master had to hand-nudge via an exact kitty `title:^void$` match, and once void exited, master misread the empty window. Trying to route the NEXT fix (muther's directed-message-wake follow-on) THROUGH void was itself blocked by void being unwakeable — the gap blocked fixing the gap.
- **Root-cause (audit-grade):** the daemon's wake is 100% TITLE-discovery. `_discover_roster_windows` scans kitty windows and DROPS any whose title doesn't resolve to a known role via `infer_role_from_name` (roster_recovery.py:290–291 `if not role: continue`). void's window title `void` isn't role-shaped → dropped → unwakeable. Three compounding facts: (a) `window_id` registration EXISTS (session_start.py POSTs it) but is GATED on `KHIMAIRA_ROSTER_SLOT` — standalone sessions skip it; (b) the `/slot` endpoint persists `roster_slot` only, NOT a retrievable `session_id→window_id`; (c) the wake (`_process_window`) consumes ONLY title-discovered windows — no identity lookup exists anywhere. So task #16 ("move liveness off title-match onto session-id") was finished for the LIVENESS read but NEVER for the WAKE path. [audit-grade: read discovery + wake + hook + slot paths end-to-end 2026-06-17.]
- **Why the test missed it:** every wake test + the watchdog's own live wake-proof used ROLE-TITLED windows (`agent-1`, the throwaway `void-actuator-test` seat) — exactly what title-discovery finds. None exercised "a legitimately-roled chat-member whose WINDOW title isn't role-shaped." Green against the premise that every wakeable session has a role-shaped window title — false for any ad-hoc / standalone-named session.
- **Catching-test:** wake-by-session-IDENTITY — persist `session_id→window_id`, register it unconditionally (whenever `$KITTY_WINDOW_ID` is present, not slot-gated), and UNION registered windows into `_process_window`'s set; then assert the daemon wakes an owing session by its REGISTERED window_id even when its title is non-role-shaped (title=`void`). **FORWARD** — 3-component fix (persist + register-unconditionally + union-into-wake); the union is additive so the title path stays and the live watchdog is untouched. Spec: `tasks/wake-by-session-id/SPEC.md`.

### catchup-staleness-cap-narrower-than-wake-latency [test-state-coverage · detector-premise]
- **Symptom (live):** muther's roster repeatedly went idle while owing in-flight work; directed dispatches + gate consults to idle critic/verifier "never reached them" and ONLY a manual kitty nudge woke them. Reported (inspection-grade) as "SSE-deafness / incomplete server.py migration" — a delivery bug.
- **Root-cause (audit-grade):** delivery is NOT broken — messages persist to JSONL, directed msgs post a durable inbox notice on dead-SSE, and a turn-time catch-up poll (`_poll_missed_chat_events`, UPS hook) replays missed events. The real bug is a TEMPORAL interaction: the catch-up poll dropped any message older than a **10-min** age cap (`user_prompt_submit.py:976,1018`), but `roster_recovery` wake latency stretches past it (`_IDLE_MIN_S=300` + `_COMPACT_COOLDOWN_S=300` + `_WIP_THRESHOLD_S=900`). So an unseen dispatch waiting >10 min was INVISIBLE to the auto-poll even when a wake finally landed — the agent woke, polled, saw nothing fresh, and re-idled. The "suppression is lossless" comment (`chats.py:4181`) was true only WITHIN the turn-gated 10-min window. Compounding: undirected broadcasts to idle-consult roles (architect/analyst/critic/verifier) are wake-suppressed with NO durable notice, so an addressed-but-undirected consult to a dead-SSE seat had no safety net at all. [audit-grade: read delivery + catch-up + wake paths end-to-end 2026-06-18; muther's "migration" mechanism FALSIFIED — the named code doesn't exist.]
- **Why the test missed it:** `test_poll_missed_chat_events_*` built messages with FRESH timestamps (1–2 min old), so the 10-min age cap NEVER triggered in any test. The fixture's temporal state didn't match production (an idle-wake target's message is OLD by the time a turn fires). Same shape as `owed-verdict-storm`: the dead premise is the TEST FIXTURE's state (message age), not a field — green against the implicit premise "missed messages are recent."
- **Catching-test:** L-unit boundary on the age dimension — with a per-chat watermark present (proves unseen), a 20-min-old msg MUST surface (the watermark `&since=` already bounds to unseen, so age must not re-drop it); cold-start (no watermark) still age-bounds replay so a fresh session doesn't dump a day of history. Plus a chats.py test: an addressed-but-undirected msg to an SSE-dead idle-consult seat posts a durable notice; unaddressed chatter does not (anti-spam). **ADDED** (test_user_prompt_submit.py +2, test_chats.py +3; Fix B decouples the cap from wake latency via watermark-gating, Fix C adds the addressed-consult notice). Spec: `tasks/sse-deaf-idle-wake/SPEC.md`. General lesson: any "lossless catch-up" claim must be tested at the temporal boundary that the recovery path can actually hit, not just on fresh state.

### watchdog-discovery-drops-prefixed-roster-workers [test-state-coverage · detector-premise]
- **Symptom (live):** the daemon's roster_recovery watchdog has injected ZERO wakes all-time for any prefixed roster — muther's entire `muther-*` roster (and any `jp-*` worker) could only be woken by manual kitty nudges. Reported by muther as "daemon auto-wake too slow/selective; never woke critic-1/tracker-1."
- **Root-cause (audit-grade):** `_discover_roster_windows` resolves each kitty window's role via `infer_role_from_name`, which only strips a trailing `-<n>` then checks `_VALID_ROLES`. `infer_role_from_name("muther-critic-1")` → strips `-1` → `"muther-critic"` → not a registry role → **None**. Discovery's `if not role: continue` (roster_recovery.py:290) then drops EVERY prefixed-roster worker window → `_discover_roster_windows` returns 0 → `_process_window` never runs → no obligation is ever checked, no wake ever fires. Confirmed live 2026-06-18: 0 windows discovered for the muther roster though all 10 seats were in `active_roster_member_ids()` and `muther-tracker-1` had a live pending obligation + `unconsumed_chat=True`. Only bare names (`critic-1`) and registered prefixed LEADS (`jp-frontend-lead`, which have their own themis yamls) resolved. [audit-grade: live `_discover_roster_windows()` + `_kitty('ls')` + `infer_role_from_name` truth-table probe.]
- **Why the test missed it:** `test_normalizes_prefixed_names` asserted only `not w["role"].endswith("-1")` — a VACUOUS invariant that held even when the window was dropped to role=None (the loop body simply didn't execute for dropped windows). The test's comment even claimed "jp-agent-1 → role=jp-agent" — aspirational; the real result was None (dropped). No test asserted the discovered ROLE VALUE for a prefixed worker, nor that the window survived discovery at all. The fixture covered the registered-lead case implicitly but never a plain prefixed worker (`muther-critic-1`).
- **Catching-test:** assert the exact resolved role for prefixed WORKERS — `muther-critic-1 → "critic"`, `muther-agent-2 → "agent"`, `jp-agent-1 → "agent"` — AND that a registered lead keeps its prefix (`jp-frontend-lead-1 → "jp-frontend-lead"`). Fix: prefix-tolerant resolver — try `infer_role_from_name(nm)`, then strip leading `<prefix>-` segments and retry. **ADDED** (test_roster_recovery.py: strengthened test_normalizes_prefixed_names to assert exact roles + new test_resolves_arbitrary_prefix_worker_roles; live-verified discovery 0→8). General lesson: a "normalizes X" test must assert the NORMALIZED VALUE, not a weak property that survives the item being silently dropped — a dropped item satisfies almost any per-item invariant vacuously.

### create-room-stores-member-roles-keys-unresolved [mock-vs-schema · contract]
- **Symptom (live):** a session added to a chat via `create_room(members=["livyatan"], member_roles={"livyatan":"agent"})` had a granted role yet was hard-blocked from EVERY write/commit on its first tool call — Themis #61 "known chat member but role cannot be resolved." `chat_grant_role` (UUID-keyed) cleared it. Observed live 2026-06-19 (the SLM-build onboarding).
- **Root-cause (audit-grade):** `create_room` (chats.py:866-874) and `POST /chats` (api/chats.py:1619-1625) persist the caller-passed `member_roles` dict VERBATIM — they never resolve its KEYS name→UUID, even though `member_session_ids` ARE resolved via `_resolve_or_uuid` (chats.py:829). So a name-keyed call stores `member_roles={"livyatan":"agent", "<creator-uuid>":"master"}` — the member's role under a NAME key while the members dict is correctly UUID-keyed. Themis `_resolve_role_from_jsonl` looks up `member_roles.get(sid)` by session UUID (themis.py:176) → miss → Layer-2 (not creator) miss → Layer-3 name-inference ("livyatan" not role-shaped) miss → Layer-4 fail-closed `_UNRESOLVABLE` → IN-UNRESOLVABLE block. [audit-grade: master observed the returned room.meta carrying the literal name key alongside a UUID-keyed members dict; the resolution path was read end-to-end.]
- **Why the test missed it:** create_room / role-resolution tests construct `member_roles` already UUID-keyed (the resolved shape Themis expects) — the green path. No test passes NAME-keyed `member_roles` through `create_room` and asserts the stored keys are UUIDs / the role resolves. The test's dict-key shape (UUID) diverged from a real ad-hoc caller's (name). The asymmetry is hidden: members-dict key resolution is exercised, member_roles key resolution is assumed. (The bootstrap-roster SKILL passes UUID keys so it's safe; direct ad-hoc `create_room` usage is the unguarded caller.)
- **Catching-test:** L1 real-producer→consumer — `create_room(members=[name], member_roles={name: "agent"})`, then assert (a) stored `room.meta.member_roles` keys are ALL UUIDs (no name keys survive) and (b) `resolve_session_role(member_uuid) == "agent"` WITHOUT any `chat_grant_role`. Fix: resolve `member_roles` keys name→UUID in `create_room` + `POST /chats`, symmetric with `member_session_ids`. **FORWARD** — hardening task filed: `tasks/create-room-resolve-member-roles-keys/SPEC.md`. SECONDARY ergonomic gap (same incident): a role-unbound member hard-blocks ALL writes only AFTER accept, with no onboarding-time signal — `invite()`/`create_room` should warn (or grant role pre-accept) when a non-role-named roster member ends up role-unbound. General lesson: when a producer resolves names→UUIDs for ONE field of a record, every sibling field keyed by the same identity must be resolved too — a half-resolved record passes any test that hand-builds the fully-resolved shape.

### wake-actuator-titlematch-misses-kitty-decorated-title [producer-mechanism · environment-config]
- **Symptom (live):** after the wake-by-session-identity fix landed (the union DID add `livyatan`'s window to the wake set, gate fully satisfied — `pending_task=True`, `obligations=[task]`, `idle>300s`, `get_session_window=978`), EVERY auto-wake on window 978 aborted for HOURS (10:25–11:00, every 60s sweep) with "TOCTOU mismatch — injected/nonce text not found in buffer". Other windows (969/970/972) woke fine in the same sweeps. A nonce + whole-buffer-scan fix (deployed mid-incident) did NOT close it; a poll-the-readback fix (1.8s) did NOT close it either — both treated a presumed readback/timing symptom. livyatan stayed unwakeable by the daemon; only a raw `--match id:978` hand-nudge landed.
- **Root-cause (audit-grade):** the send-text never reached the window — the READBACK was never the problem. The union path passes the CLEAN session name as `window_title` (`raw_name = name_by_sid.get(sid)` → `"livyatan"`, roster_recovery.py:522), but kitty decorates the LIVE window title with a dynamic activity marker — real title `✳ livyatan` (idle-activity), and it flips to `⠂ livyatan` (braille thinking-spinner) when the session is processing. `_title_match_arg` anchors the match as `title:^livyatan$`, which cannot match `✳ livyatan`. Proven live: `kitty @ ls --match 'title:^livyatan$'` → "No matching windows"; `kitty @ send-text --match 'title:^livyatan$'` → **rc=0 (a SILENT no-op, not an error)** → keystrokes go nowhere; the id-based nonce readback correctly finds nothing → abort (hence "TOCTOU mismatch", never "send-text failed"). Role-named windows (969/970/972) escaped ONLY because title-discovery carried their FULL decorated title as `raw_name`; the marker can still flicker between discovery and inject for them too. [audit-grade: live `kitty @ ls`/`send-text`/`get-text` probes on window 978 2026-06-20 — falsified my own prior timing & readback-scope hypotheses by observing send-text land via id-match but not via the anchored title-match.]
- **Why the test missed it:** `TestInjectTextAndSubmit` mocks `_kitty` entirely and forces `_count_title_windows → 1` (autouse fixture), with a stateful echo-mock that ECHOES the inject (incl. nonce) into the get-text screen — so the title-match ALWAYS "resolves" and the inject ALWAYS "lands". No test ever ran a real kitty matcher against a decoration-prefixed/flickering title, nor asserted what happens when the anchored title matches ZERO live windows. Green against the premise "the window_title the daemon holds == kitty's live title" — false the moment kitty adds its dynamic marker. Same family as the sibling `wake-targets-window-by-title-not-session-identity`: the identity read was fixed but the ACTUATION still trusted a title string that production decorates.
- **Catching-test:** branch the actuator on how many live windows the title matches — `n==1` → title-match; `n==0` → the decorated/flickering title matches no live window, fall back to the `window_id` resolved THIS sweep (fresh, not stale; the nonce TOCTOU still guards a genuinely-dead id); `n>1` → reap+abort. **ADDED** (`test_falls_back_to_id_when_title_matches_no_window`: `_count_title_windows → 0` + title provided → asserts send-text uses `--match=id:` not `title:` and still submits; plus `test_submits_when_nonce_echoes_late` for the poll path). Live-verified: id-match send-text lands the nonce on window 978 where title-match silently no-ops. Commits: `ce7af6e` (id-fallback), `5e5f9c5`/poll (defense-in-depth). The manual `/khimaira-nudge` skill carried the IDENTICAL bug (anchored `title:^name$` on a decorated title) — fixed in lockstep (select-by-name, actuate-by-fresh-id, readback-confirm). General lesson: when a runtime (kitty/tmux/a window manager) DECORATES the very identifier you match on, an anchored exact-match silently no-ops — match on a normalized form for SELECTION but actuate by a stable handle (the id) you resolved in the same pass, and treat a readback (not an exit code) as the only success signal.

## 3. Forward test-strategy (layered by seam-class)
- **L1 — real-producer→consumer e2e** per surface (the real event/payload a producer emits, not an injected dict): catches entry-path, producer-mechanism, producer→payload seams.
- **L2 — real-store integration** (real chat JSONL / real DB, not a mocked store): catches logic seams the mock never runs.
- **L3 — contract checks** (every mocked field/column/symbol vs the real schema/runtime shape): catches mock-vs-schema + detector-premise-vs-runtime seams.
- **L4 — live-runtime exercise** (Specter-in-CI for UI; real-daemon for orchestration): catches environment-config + frontend seams.
- **L0 (meta) — assert-it-runs / assert-it-enforces gate**: the enforcement mechanism must EXECUTE (a test) or INTERCEPT (a governance hook) for real on the relevant path — assert `executed>0` / `blocked==true`, not just `failures==0` / `rule-in-config`. Catches the recursive escape where a catching-test silently no-ops (wrong driver, unreachable store) OR a structural rule is dormant (hook absent in some session class).

**Seam-class vocabulary** (extend as new classes appear): `entry-path` · `producer→event-payload` · `producer-mechanism` · `mock-vs-schema` · `SQL-logic` · `environment-config` · `contract` · `frontend-render` · `enforcement-premise` (a governance/detector rule whose enforcement mechanism — hook, interceptor — is absent in some runtime/environment, so the rule is present-in-config but dormant in practice) · `detector-premise-vs-runtime` (a detector keys on an input premise — field/string/symbol — that real runtime data never produces).
